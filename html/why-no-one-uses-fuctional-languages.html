Why no one uses functional languages (1998)

Why no one uses functional languages is a (very concise and easy to read 1) paper written by Philip Wadler.

Although it talks about functional languages in particular, I believe many of its points also apply to other, non-functional languages.

The article lays out a set of points where functional languages fall short, thus hurting their adoption both by hobbyists and the industry.

The reasons

"Not playing well" with other languages

Today's programs often leverage code written in other languages through the use of Foreign Function Interfaces.

Popular languages like Python are often described as being "glue" to other programs written using other languages.

This description is not entirely without merit, considering that some of its major libraries like numpy, SciPy and even its standard library (!) are implemented in C.

The problem with some languages is that reaching out to foreign code breaks their model, or it negates some of their advantages.

For this reason some languages prefer to keep to themselves and their little worlds.

The ecosystem and libraries (or lack thereof)

Lack of libraries increase the chances that people working with that particular language will need to, sooner or later, implement something that probably would be provided as a library in another, more popular, language. Be it GUI libraries, ML ones, https, etc…

This increases the programmers workload and the time needed to complete a project, which can entirely outweight the advantages given by using that language in the first place. Which can lead to companies opting for languages with a higher number of libraries.

Availability in different architectures/OSes

Not all software is intended to run on a browser or on a major OS.

Maybe the program you're writing is intended to run on some old SPARCstation, perhaps its intended to run on a MIPS router, or maybe its for an IBM Mainframe.

Many smaller languages don't have the means nor the manpower to port their implementations to these platforms. And while some of the above examples may not be very convincing, consider things like embedded devices (like your home appliances), financial institutions, healthcare, and many more.

How easy it is to get started

Is the language and its tooling easy to obtain, install and use?

What about learning resources? Are those widely available?

Is the documentation up to date? (if it even has that)

Support

Stability

Is it a new language? Are newer versions backwards compatible? How frequent are breaking changes?

How well does it work on a resource constrained system

Are the executable produced too big?

How is the memory usage?

Tooling

This one is less important now than it was with the advent of things like LSP and DAP.

Nowadays a language server can take a language pretty far in terms of tooling.

Still, one frequent question asked about newer languages is if they provide a language server.

The non-reasons

The paper also gives 2 non-reasons for not using functional languages, non-reasons as, the author doesn't believe these are the reasons that make programmers choose other languages.

Performance

The 90s, when this paper was published, was a era of quick technological process, especially in terms of hardware power.

Lisp, the first functional programming language, saw limited use outside its niche, due to hardware limitations.

Nowadays, especially with the constant improvements to functional programming, I'd argue most software doesn't have the need for C-like performance.

Ignorance

The author argues that the above points are a stronger reason for not choosing functional prorgamming.
